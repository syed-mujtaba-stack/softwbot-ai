# 08 — User Interface

---

## Executive Summary

This document defines the UI philosophy, design patterns, layout system, component specifications, page-level designs, state management for UI, and accessibility requirements for every screen in SoftwBot AI. The UI follows a glassmorphism-inspired, minimal, professional, animated, and fully accessible design language.

---

## Purpose

The UI documentation ensures consistent, high-quality user experiences across all pages. It serves as the source of truth for frontend developers and designers.

---

## Design Philosophy

| Principle | Description |
|-----------|-------------|
| **Glassmorphism** | Frosted-glass panels with subtle blur, transparency, and layered depth |
| **Minimal** | Clean layouts, generous whitespace, focused content |
| **Professional** | Business-grade aesthetics, not playful or childish |
| **Animated** | Smooth transitions, micro-interactions, purposeful motion |
| **Accessible** | WCAG 2.1 AA compliant, keyboard-navigable, screen-reader friendly |
| **Responsive** | Works flawlessly from 320px to 2560px |

## Component Specifications

### Glassmorphism Card

```css
background: rgba(255, 255, 255, 0.7);
backdrop-filter: blur(12px);
border: 1px solid rgba(255, 255, 255, 0.2);
border-radius: 16px;
box-shadow: 0 4px 6px -1px rgba(0, 0, 0, 0.1);
```

### Page Transitions

- Use Framer Motion `AnimatePresence` for route transitions
- Enter: fade in + slide up 20px, duration 200ms
- Exit: fade out, duration 150ms
- Staggered list items: 50ms delay between items

### Loading States

Every data-dependent page must have:
1. **Skeleton loader** — Matches layout of loaded content
2. **Spinner** — For inline loading (buttons, small areas)
3. **Progress bar** — For known-duration operations (uploads, processing)
4. **Empty state** — When no data exists (illustration + CTA)
5. **Error state** — When loading fails (error message + retry button)
6. **Success state** — After successful action (toast + updated data)

## Page Specifications

### Dashboard Overview

**Purpose:** At-a-glance view of workspace health

**Components:**
- 4 metric cards (Total Bots, Active Conversations, Messages Today, Leads Captured)
- Activity feed (last 20 events)
- Quick actions (Create Bot, View Conversations, Broadcast)
- Performance chart (messages over time, line chart)
- Bot status grid (each bot with status indicator)

**Actions:**
- Click metric card → filtered view
- Click activity item → navigate to relevant page
- Quick action buttons → respective pages

### Conversations Inbox

**Purpose:** View and manage all WhatsApp conversations

**Components:**
- Conversation list (left panel, scrollable)
  - Contact avatar, name, last message preview, time, status badge, unread count
- Chat thread (center panel)
  - Message bubbles (AI: left/blue, Human: left/gray, User: right/green)
  - Timestamps, read receipts, confidence scores
- Contact sidebar (right panel, collapsible)
  - Contact info, tags, lead score, notes, history
- Top bar: Search, filter dropdown, bulk actions

**States:**
- Loading: Skeleton conversation list
- Empty: "No conversations yet. Messages will appear here when customers reach out."
- Error: "Unable to load conversations. Please try again."
- Success: Full conversation list with real-time updates

### Bot Architect

**Purpose:** Create bots via natural language conversation

**Components:**
- Chat interface (center, message thread)
- AI avatar + typing indicator
- Text input with send button
- Generated config preview panel (right side, collapsible)
  - System prompt preview
  - Personality card
  - Automation rules list
  - Knowledge base suggestions
  - Model recommendation
- Progress indicator (steps: Describe → Features → Configure → Preview → Deploy)
- Deploy button (primary CTA)

## Animation Guidelines

| Interaction | Animation | Duration | Easing |
|------------|-----------|----------|--------|
| Page transition | Fade + slide up | 200ms | ease-out |
| Card hover | Scale 1.02 + shadow | 150ms | ease-in-out |
| Modal open | Fade + scale from 0.95 | 200ms | ease-out |
| Modal close | Fade out | 150ms | ease-in |
| Toast enter | Slide in from right | 300ms | spring |
| Toast exit | Slide out to right | 200ms | ease-in |
| Sidebar toggle | Width transition | 200ms | ease-in-out |
| Button click | Scale 0.97 | 100ms | ease-in-out |
| List item enter | Fade + slide down | 200ms | ease-out, staggered |
| Dropdown open | Fade + slide down 8px | 150ms | ease-out |
| Skeleton pulse | Opacity pulse | 1500ms | ease-in-out, infinite |
| Typing indicator | Three dots bounce | 600ms | ease-in-out, infinite |

## Dark Mode / Light Mode

- Toggle in Settings → Appearance
- System preference detection via `prefers-color-scheme`
- Toggle triggers CSS variable swap (no flash on load)
- Persisted in localStorage
- All components must support both modes

## Accessibility Checklist

- [ ] All images have `alt` text
- [ ] All buttons have accessible labels
- [ ] All form inputs have associated labels
- [ ] All modals trap focus
- [ ] All interactive elements are keyboard accessible
- [ ] Skip navigation link present
- [ ] ARIA landmarks used (main, nav, aside)
- [ ] Live regions for dynamic content updates
- [ ] Error messages associated with form fields
- [ ] Color is never the only indicator of state

---

## Developer Notes

- Use Shadcn UI components as base — extend, don't replace
- Use Framer Motion for all animations (not CSS transitions for complex ones)
- Tailwind CSS v4 for all styling — no CSS modules, no inline styles
- All components must accept `className` prop for composition
- Use `cn()` utility for conditional class merging
- Test with keyboard-only navigation regularly
- Test with screen reader (VoiceOver / NVDA) before each release
